# HALOS Provenance Model v0.2

**Status:** Active
**Extends:** HALOS Provenance Spec v0.1
**Schema:** [`../schema/halos-provenance-v0.2.schema.json`](../schema/halos-provenance-v0.2.schema.json)
**Principles anchor:** HALOS-CORE-1, HALOS-CORE-3, HALOS-CORE-4

> v0.2 is active and normative. To propose changes or additions for v0.3, open a proposal in the [halos proposals directory](https://github.com/northharbor-dev/halos/tree/main/proposals).

---

## Motivation

The v0.1 spec records provenance as a flat document: one artifact, one human author, one list of AI contributions. This is sufficient for most cases.

However, some artifacts have complex histories:

- An artifact derived from multiple parent artifacts
- Multiple humans contributing at different stages
- AI used at some stages but not others
- A review chain across multiple reviewers
- Decisions made during the process that shape the outcome
- AI suggestions that were accepted, modified, or rejected

The v0.2 model addresses this by representing provenance as a **graph** of Entities, Activities, and Relationships — closely aligned with the W3C PROV model and the terminology defined in [`../terminology.md`](../terminology.md).

v0.2 also introduces first-class support for **decision provenance**, **human–AI interaction semantics**, **lightweight responsibility modeling**, and **policy evaluation traces**.

---

## New in v0.2: Task Type and Techniques

v0.2 introduces two optional fields — `taskType` and `techniques` — that describe *what cognitive task the AI performed* and *what named algorithms or methods it employed*. These fields appear in three places:

- `ai_assistance[]` — the flat AI contribution list
- `graph.activities[]` — the graph activity model
- `decisions[].aiInputs[]` — AI contributions to decisions

### Why

The existing `role` field (generation, drafting, editing, review, assistance) and `activity.type` (create, generate, edit, suggest, transform...) describe the AI's *relationship to the artifact* — how its output fits into the artifact lifecycle. They do not capture:

1. **The cognitive task** the AI performed — classifying, researching, summarizing, predicting, extracting, orchestrating, etc. This is distinct from the artifact-lifecycle role. An AI can perform "research" (taskType) in a "generation" (role) capacity.
2. **The specific techniques** the AI used — Monte Carlo simulation, retrieval-augmented generation, chain-of-thought prompting, Bayesian inference, etc. Knowing the technique matters for reproducibility, auditability, and trust calibration.

**Software development task types:**

## Core Concepts

### Entity

A thing that exists and can be described. In HALOS, entities include:

- **Artifact** — a digital output (code, document, dataset, etc.)
- **Person** — a human author, reviewer, or contributor
- **Agent** — an AI model or tool
- **Organization** — a team, company, or institution
- **Policy** — a governance policy or standard

### Activity

An action performed over time that used or generated one or more entities. Activities include:

- `create` — initial authoring of an artifact
- `generate` — AI generation of an artifact or portion
- `edit` — modification of an existing artifact
- `review` — a human evaluation step
- `approve` — a formal sign-off
- `derive` — producing an artifact from another
- `decision` — a recorded decision point (see [Decision Provenance](#decision-provenance))
- `suggest` — an AI suggestion offered for human consideration
- `transform` — an AI transformation of existing content
- `other` — any activity not covered above

### Relationship

A typed connection between entities and/or activities.

#### Recommended Relationship Types (non-normative)

The following relationship types are aligned with W3C PROV semantics. They are recommendations — implementations may extend this set for domain-specific needs.

| Relationship | Connects | Meaning |
|---|---|---|
| `wasGeneratedBy` | Artifact → Activity | The artifact was produced by this activity |
| `wasAttributedTo` | Artifact → Person/Agent | The artifact is attributed to this entity |
| `used` | Activity → Artifact | The activity used this artifact as input |
| `wasAssociatedWith` | Activity → Person/Agent | This entity participated in the activity |
| `wasDerivedFrom` | Artifact → Artifact | The artifact was derived from this parent |
| `actedOnBehalfOf` | Agent → Person | The agent acted under direction of this person |
| `wasInformedBy` | Activity → Activity | This activity was informed by another activity |
| `hadRole` | Entity → Activity | The entity had a specific role in the activity |

For implementations that prefer simpler, more readable relationship names, the following aliases are also recognized:

| Alias | Equivalent to |
|---|---|
| `authored_by` | `wasAttributedTo` |
| `reviewed_by` | `wasAssociatedWith` (where activity type is `review`) |
| `approved_by` | `wasAssociatedWith` (where activity type is `approve`) |
| `generated_with` | `wasAssociatedWith` (where entity type is `agent`) |
| `derived_from` | `wasDerivedFrom` |
| `influenced_by` | `wasInformedBy` |

Aliases are syntactic convenience — they carry the same semantics as their PROV-aligned equivalents.

## Record Structure (v0.2)

A v0.2 provenance record extends v0.1 with optional `graph`, `decisions`, and `policyEvaluations` fields.

```json
{
  "halos_version": "0.2-draft",
  "artifact": { ... },
  "human_author": { ... },
  "ai_assistance": [ ... ],
  "timestamp": "...",

  "graph": {
    "entities": [ ... ],
    "activities": [ ... ],
    "relationships": [ ... ]
  },
  "decisions": [ ... ],
  "policyEvaluations": [ ... ]
}
```

All v0.2-specific fields (`graph`, `decisions`, `policyEvaluations`) are optional. A v0.2 record without these fields is equivalent to v0.1.

### Extended `ai_assistance` Fields

v0.2 adds two optional fields to each `ai_assistance` entry:

| Field | Type | Required | Description |
|---|---|---|---|
| `taskType` | string | no | The cognitive task the AI performed (see [Recommended Task Types](#recommended-task-types-non-normative)) |
| `techniques` | array of strings | no | Named algorithms or methods the AI employed (see [Recommended Techniques Vocabulary](#recommended-techniques-vocabulary-non-normative)) |

Example:

```json
{
  "model": "claude-sonnet-4-6",
  "provider": "Anthropic",
  "role": "assistance",
  "taskType": "classifying",
  "techniques": ["retrieval-augmented-generation", "chain-of-thought"],
  "description": "Classified documents against internal taxonomy using RAG pipeline"
}
```

## Graph Model

### `graph.entities`

Each entity:

```json
{
  "id": "urn:entity:jane-smith",
  "type": "person",
  "label": "Jane Smith",
  "identifier": "https://github.com/janesmith",
  "role": "author",
  "delegatedBy": "urn:entity:team-lead"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Stable local identifier for this entity within the record |
| `type` | string | yes | One of: `artifact`, `person`, `agent`, `organization`, `policy` |
| `label` | string | yes | Human-readable name |
| `identifier` | string | no | External URI or handle |
| `role` | string | no | Role in the context of this record (see [Roles](#roles)) |
| `delegatedBy` | string | no | ID of the entity that delegated authority to this entity |

#### Roles

Roles are recommended values, not a closed enum. Common roles include:

- `author` — primary creator
- `reviewer` — evaluator of the artifact
- `approver` — formal sign-off authority
- `contributor` — participant in creation
- `operator` — person who directed or supervised an AI agent

Implementations may define additional roles. The intent is to capture **who did what**, not to enforce rigid access control.

### `graph.activities`

Each activity:

```json
{
  "id": "urn:activity:draft-v1",
  "type": "create",
  "startedAt": "2026-03-20T10:00:00Z",
  "endedAt": "2026-03-20T11:30:00Z",
  "description": "Initial draft of the document",
  "taskType": "scaffolding",
  "techniques": ["retrieval-augmented-generation"],
  "interaction": {
    "type": "generation",
    "humanResponse": "modified",
    "responseNotes": "Accepted overall structure, rewrote conclusion section"
  }
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Stable local identifier |
| `type` | string | yes | Activity type (see [Activity Types](#activity-types)) |
| `startedAt` | string | no | ISO 8601 start time |
| `endedAt` | string | no | ISO 8601 end time |
| `description` | string | no | Free-text description |
| `taskType` | string | no | The cognitive task the AI performed (see [Recommended Task Types](#recommended-task-types-non-normative)) |
| `techniques` | array of strings | no | Named algorithms or methods employed (see [Recommended Techniques Vocabulary](#recommended-techniques-vocabulary-non-normative)) |
| `interaction` | object | no | Human–AI interaction metadata (see [Interaction Semantics](#humanai-interaction-semantics)) |

#### Activity Types

| Type | Description |
|---|---|
| `create` | Initial authoring of an artifact |
| `generate` | AI generation of content |
| `edit` | Modification of existing content |
| `review` | Human evaluation |
| `approve` | Formal sign-off |
| `derive` | Producing an artifact from another |
| `decision` | A recorded decision point |
| `suggest` | AI suggestion offered for human consideration |
| `transform` | AI transformation of existing content |
| `other` | Any other activity |

### `graph.relationships`

Each relationship:

```json
{
  "type": "wasAttributedTo",
  "from": "urn:entity:my-document",
  "to": "urn:entity:jane-smith",
  "notes": "Primary author"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | string | yes | Relationship type (see table above) |
| `from` | string | yes | ID of the source entity or activity |
| `to` | string | yes | ID of the target entity or activity |
| `notes` | string | no | Optional context |

## Decision Provenance

Decisions are first-class concepts in v0.2. A decision records a point where a human made a meaningful choice that shaped the artifact — with or without AI input.

Conceptually, a decision is a **specialized activity** — it is something that happened at a point in time, involved agents, and influenced the artifact. This keeps the model consistent: decisions follow the same entity–activity–relationship semantics as all other provenance events.

Decisions can be expressed in two ways:
1. **Graph activity (preferred)** — as an activity of type `decision` within the graph, connected to agents and artifacts via relationships. This is the canonical representation and allows the richest modeling.
2. **Flat array (convenience)** — as entries in the top-level `decisions` array, for simpler records that don't need a full graph. This is syntactic sugar — each entry represents the same concept as a `decision` activity.

### `decisions` Array

```json
{
  "decisions": [
    {
      "id": "urn:decision:arch-choice-001",
      "description": "Selected event-driven architecture over request-response",
      "responsible": "Jane Smith",
      "responsibleId": "https://github.com/janesmith",
      "timestamp": "2026-03-20T14:00:00Z",
      "context": "Needed to handle bursty traffic patterns with minimal latency",
      "aiInputs": [
        {
          "model": "claude-sonnet-4-6",
          "provider": "Anthropic",
          "contribution": "Compared three architecture patterns with trade-off analysis"
        }
      ],
      "rationale": "Event-driven better supports our scale requirements and team expertise",
      "outcome": "accepted"
    }
  ]
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Stable identifier for the decision |
| `description` | string | yes | What was decided |
| `responsible` | string | yes | Name of the accountable human decision-maker |
| `responsibleId` | string | no | External identifier for the decision-maker |
| `timestamp` | string | yes | ISO 8601 datetime of the decision |
| `context` | string | no | Circumstances or constraints that framed the decision |
| `aiInputs` | array | no | AI contributions that informed the decision |
| `aiInputs[].model` | string | yes | AI model name or identifier |
| `aiInputs[].provider` | string | no | Organization providing the model |
| `aiInputs[].contribution` | string | yes | What the AI contributed to the decision |
| `aiInputs[].taskType` | string | no | The cognitive task the AI performed for this input |
| `aiInputs[].techniques` | array of strings | no | Named algorithms or methods used |
| `rationale` | string | no | Why this decision was made |
| `outcome` | string | no | Result of the decision (e.g., `"accepted"`, `"deferred"`, `"rejected"`) |

### Decision as Graph Activity

When using the full graph model, a decision can also be represented as an activity:

```json
{
  "graph": {
    "activities": [
      {
        "id": "urn:activity:arch-decision",
        "type": "decision",
        "description": "Selected event-driven architecture",
        "startedAt": "2026-03-20T14:00:00Z"
      }
    ],
    "relationships": [
      {
        "type": "wasAssociatedWith",
        "from": "urn:activity:arch-decision",
        "to": "urn:entity:jane-smith"
      },
      {
        "type": "wasInformedBy",
        "from": "urn:activity:arch-decision",
        "to": "urn:activity:ai-analysis"
      }
    ]
  }
}
```

Both representations are valid. The `decisions` array is simpler for flat records; the graph representation allows richer relationship modeling.

## Human–AI Interaction Semantics

v0.2 introduces structured metadata for how humans and AI systems interact during artifact creation. This captures not just *what* the AI did, but *how the human responded* to AI output.

### Interaction Object

The `interaction` field on activities records the nature of the AI contribution and the human response:

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | string | yes | Nature of AI involvement |
| `humanResponse` | string | no | How the human responded to the AI output |
| `responseNotes` | string | no | Details about modifications or reasoning |

#### Recommended Interaction Types (non-normative)

The following values are recommendations to promote consistency across implementations. They are **not** a closed enum — implementations may extend them as needed.

**AI contribution types** (`interaction.type`):

| Type | Description |
|---|---|
| `suggestion` | AI offered an option; human decided whether to use it |
| `generation` | AI produced content from scratch |
| `transformation` | AI modified or restructured existing content |
| `analysis` | AI analyzed content and provided findings |
| `review` | AI reviewed content for issues or improvements |

**Human response types** (`interaction.humanResponse`):

| Value | Description |
|---|---|
| `accepted` | Human accepted the AI output without changes |
| `modified` | Human accepted with modifications |
| `rejected` | Human rejected the AI output |
| `partial` | Human used some parts and discarded others |
| `pending` | AI output has not yet been reviewed by a human |

### Example: AI Suggestion Accepted with Modifications

```json
{
  "id": "urn:activity:code-gen-auth-module",
  "type": "generate",
  "description": "AI generated authentication module",
  "interaction": {
    "type": "generation",
    "humanResponse": "modified",
    "responseNotes": "Accepted structure and logic; replaced hardcoded values with env vars, added rate limiting"
  }
}
```

## Policy Evaluation Trace

v0.2 supports recording the results of policy evaluations — automated or manual checks that an artifact or process was assessed against a policy.

This feature **records results only**. It does not implement policy logic, enforce rules, or prescribe what policies should exist.

### `policyEvaluations` Array

```json
{
  "policyEvaluations": [
    {
      "id": "urn:eval:security-review-001",
      "policy": "OWASP Top 10 Compliance",
      "policyVersion": "2025",
      "evaluator": "snyk-scanner",
      "evaluatorType": "tool",
      "result": "pass",
      "timestamp": "2026-03-20T16:00:00Z",
      "notes": "No critical or high vulnerabilities found"
    },
    {
      "id": "urn:eval:halos-compliance-001",
      "policy": "HALOS Principles v1.0",
      "evaluator": "Jane Smith",
      "evaluatorType": "human",
      "result": "pass",
      "timestamp": "2026-03-20T16:30:00Z",
      "notes": "Human authorship confirmed; AI assistance properly disclosed"
    }
  ]
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Stable identifier for this evaluation |
| `policy` | string | yes | Name of the policy or standard being evaluated against |
| `policyVersion` | string | no | Version of the policy |
| `evaluator` | string | yes | Name of the person, tool, or system that performed the evaluation |
| `evaluatorType` | string | yes | One of: `human`, `tool`, `service` |
| `result` | string | yes | Outcome: `pass`, `fail`, `warning`, `error`, `skipped` |
| `timestamp` | string | no | ISO 8601 datetime of the evaluation |
| `notes` | string | no | Additional context or findings |
| `details` | object | no | Tool-specific structured output (opaque to HALOS) |

## Compatibility with v0.1

v0.2 is a **strict superset** of v0.1. This means:

1. **Every valid v0.1 record is a valid v0.2 record.** The required fields (`halos_version`, `artifact`, `human_author`, `ai_assistance`, `timestamp`) and optional fields (`review`, `governance`, `lineage`, `notes`) are unchanged.

2. **All new fields are optional.** A v0.2 record without `graph`, `decisions`, or `policyEvaluations` is structurally equivalent to v0.1.

3. **Tools that only understand v0.1 should ignore unknown fields.** The v0.1 schema uses `additionalProperties: false`, so strict v0.1 validators will reject records with v0.2 fields — but the `halos_version` field distinguishes them. Consumers should route to the appropriate schema based on `halos_version`.

4. **No v0.1 field semantics have changed.** The meaning of `human_author`, `ai_assistance`, `review`, `governance`, and `lineage` is identical. v0.2 provides richer alternatives (graph, decisions, interaction semantics) but does not deprecate the v0.1 fields.

### Migration Path

- Records authored against v0.1 need no changes to remain valid.
- To upgrade a v0.1 record to v0.2, set `halos_version` to `"0.2-draft"` and optionally add graph, decision, or policy evaluation data.
- The flat v0.1 fields and the graph model can coexist in the same record. This allows incremental adoption — start with the familiar flat structure and add graph data as needed.

## Non-Goals

To maintain focus and prevent scope creep, the following are explicitly **not** goals of HALOS:

| Non-Goal | Belongs to |
|---|---|
| Risk scoring or vulnerability assessment | AIVSS, CVSS |
| Organizational governance frameworks | NIST AI RMF, ISO/IEC 42001 |
| Policy definition or enforcement | Policy engines (OPA, Cedar) |
| Software bill of materials (SBOM) | CycloneDX, SPDX |
| Build provenance and supply chain integrity | SLSA |
| Identity verification or authentication | OIDC, SPIFFE, X.509 |
| Access control or authorization | RBAC/ABAC systems |

HALOS records **evidence of what happened** — who was involved, what AI contributed, what decisions were made. It does not evaluate, enforce, or score. Systems that do those things can consume HALOS records as input.

## Ecosystem Alignment

HALOS occupies a specific niche in the AI governance and supply chain ecosystem. It is the **human-centered provenance and accountability layer** — complementary to, not competing with, adjacent standards and frameworks.

### At a Glance

| System Category | Role | Examples |
|---|---|---|
| Risk scoring frameworks | Evaluate AI risk | AIVSS, CVSS |
| Governance frameworks | Define policies and controls | NIST AI RMF, ISO 42001 |
| Supply chain standards | Track components and build integrity | CycloneDX, SLSA, SPDX |
| Provenance ontologies | General-purpose provenance modeling | W3C PROV |
| **HALOS** | **Track human–AI collaboration and accountability** | — |

### HALOS and AIVSS

**AIVSS** (AI Vulnerability Scoring System) provides a standardized approach to scoring vulnerabilities in AI systems.

- **HALOS = evidence.** It records what happened — who was involved, what AI contributed, what decisions were made.
- **AIVSS = scoring.** It evaluates risk severity for AI-specific vulnerabilities.

HALOS provenance records can serve as **input evidence** for AIVSS assessments. For example, a HALOS record showing that AI-generated code was deployed without human review would be relevant to an AIVSS evaluation. HALOS does not score risk; it provides the factual substrate that scoring systems operate on.

### HALOS and NIST AI RMF / ISO 42001

**NIST AI RMF** and **ISO/IEC 42001** are governance frameworks that define organizational practices for responsible AI.

- **HALOS = implementation-level provenance.** It defines the structure of individual provenance records attached to specific artifacts.
- **NIST/ISO = organizational governance.** They define management systems, risk categories, and organizational processes.

HALOS operates at the artifact level, inside the processes that governance frameworks describe. An organization following NIST AI RMF might use HALOS provenance records as the mechanism for demonstrating traceability (NIST MAP, MEASURE functions). HALOS does not define governance processes; it provides the records those processes require.

### HALOS and W3C PROV

**W3C PROV** is a data model for provenance on the web, defining Entities, Activities, and Agents with formal semantics.

- **HALOS v0.2 graph model is conceptually aligned with W3C PROV.** Entities, Activities, and Relationships in HALOS correspond to prov:Entity, prov:Activity, and the PROV relationship types.
- **HALOS simplifies.** W3C PROV is a general-purpose ontology with formal RDF semantics. HALOS adopts the intuitive structure but uses plain JSON, simpler typing, and domain-specific extensions (decisions, interaction semantics) that W3C PROV does not prescribe.

Implementations that need full W3C PROV interoperability can map HALOS graph data to PROV-O or PROV-JSON representations. HALOS does not require this.

### HALOS and CycloneDX / SLSA

**CycloneDX** (SBOM standard) and **SLSA** (build provenance framework) address software supply chain integrity.

- **CycloneDX** inventories components and their metadata. HALOS provenance records can be embedded as `component.evidence` or custom properties within a CycloneDX SBOM.
- **SLSA** verifies that builds were produced by trusted systems from trusted sources. HALOS provenance can supplement SLSA attestations with human-authorship and AI-disclosure data that SLSA does not cover.

HALOS is **complementary**. CycloneDX answers "what is in this software?" SLSA answers "how was it built?" HALOS answers "who was responsible, what AI contributed, and what decisions were made?"

See [`../../mappings/cyclonedx-slsa.md`](../../mappings/cyclonedx-slsa.md) for embedding details.

## Open Questions

The following are unresolved and subject to change:

1. **Identifier namespacing** — Should entity IDs be scoped to the record, or globally addressable? Current recommendation: record-scoped with optional external identifiers.
2. **Multi-artifact records** — Should a single record describe multiple artifacts, or require one record per artifact? Current recommendation: one primary artifact per record; use graph relationships to reference related artifacts.
3. **Signing** — How should records be signed to enable verification? This is deliberately deferred — HALOS focuses on structured claims, not cryptographic proof.
4. **Decision granularity** — How detailed should decision records be? Current recommendation: capture decisions that materially shaped the artifact; avoid recording every micro-decision.

Feedback on these questions is welcome via issue.
