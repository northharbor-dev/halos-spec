# HALOS Provenance Model v0.3

**Status:** Active
**Extends:** HALOS Provenance Spec v0.2
**Schema:** [`../schema/halos-provenance-v0.3.schema.json`](../schema/halos-provenance-v0.3.schema.json)
**Principles anchor:** HALOS-CORE-1, HALOS-CORE-3, HALOS-CORE-4

> v0.3 is active and normative. To propose changes or additions for v0.4, open a proposal in the [halos proposals directory](https://github.com/northharbor-dev/halos/tree/main/proposals).

---

## Motivation

In v0.1 and v0.2, the `governance` field is a single object referencing one governance policy. In practice, artifacts routinely fall under multiple concurrent governance policies:

- A payment service governed by both an internal AI usage policy and PCI-DSS compliance requirements
- A water treatment system update governed by both an internal change management policy and a state environmental permit
- An academic paper governed by both a university AI use policy and a journal's disclosure requirements
- A government report governed by both an agency AI transparency policy and public records statutes

The v0.1/v0.2 design forces authors to either pick one policy (losing traceability to others) or concatenate multiple policies into a single string (losing structure). Neither is acceptable for compliance and audit use cases.

v0.3 addresses this by changing `governance` from a single object to an **array of policy references**.

---

## Change Summary

### `governance`: Object → Array

**v0.2 (single object):**

```json
{
  "governance": {
    "policy": "Meridian AI Usage Policy for Shared Infrastructure",
    "version": "2.1",
    "url": "https://internal.meridian.example/policies/ai-usage-shared-infra"
  }
}
```

**v0.3 (array of objects):**

```json
{
  "governance": [
    {
      "policy": "Meridian AI Usage Policy for Shared Infrastructure",
      "version": "2.1",
      "url": "https://internal.meridian.example/policies/ai-usage-shared-infra"
    },
    {
      "policy": "PCI-DSS",
      "version": "4.0",
      "url": "https://www.pcisecuritystandards.org/document_library/"
    }
  ]
}
```

Each element in the array has the same shape as the v0.2 governance object:

| Field | Type | Required | Description |
|---|---|---|---|
| `policy` | string | yes | Name or identifier of the governance policy |
| `version` | string | no | Version of the policy |
| `url` | string | no | URL to the policy document |

An empty array (`[]`) indicates that no governance policy was declared. A single-element array is equivalent to the v0.2 single-object form.

---

## Relationship to `policyEvaluations`

The `governance` array and `policyEvaluations` array serve different purposes:

| | `governance` | `policyEvaluations` |
|---|---|---|
| **Answers** | "Under what policies was this artifact produced?" | "What evaluations were actually performed?" |
| **Timing** | Declared at artifact creation | Recorded after evaluation occurs |
| **Coverage** | May include policies with no automated evaluation | Only includes policies that were actually evaluated |
| **Content** | Policy identity (name, version, URL) | Evaluation results (pass/fail, evaluator, notes) |

An artifact may list a governance policy that has no corresponding `policyEvaluation` entry — for example, a broad organizational AI ethics policy that applies but has no automated check. Conversely, a `policyEvaluation` may reference a standard (like OWASP Top 10) that is not listed in `governance` because it is a best-practice check rather than a governing policy.

## Compatibility

### Breaking Change from v0.2

This is a **breaking change**. The `governance` field type changes from `object` to `array`. Records that use the v0.2 single-object form will not validate against the v0.3 schema.

The `halos_version` field distinguishes schema versions. Consumers should route to the appropriate schema based on this field.

### All Other Fields Unchanged

All fields other than `governance` retain their v0.2 semantics and structure. The v0.3 schema is otherwise identical to v0.2.

### Migration Path

To upgrade a v0.2 record to v0.3:

1. Set `halos_version` to `"0.3"`
2. Wrap the existing `governance` object in an array:

```json
// v0.2
"governance": { "policy": "HALOS", "version": "1.0" }

// v0.3
"governance": [{ "policy": "HALOS", "version": "1.0" }]
```

3. Add any additional governance policies that apply to the artifact

For implementations that must support both v0.2 and v0.3 records, check `halos_version` and normalize:
- If `governance` is an object, treat it as a single-element array
- If `governance` is an array, use it directly

## Examples

### Single Policy (Equivalent to v0.2)

```json
{
  "halos_version": "0.3",
  "governance": [
    {
      "policy": "HALOS",
      "version": "1.0",
      "url": "https://halos.northharbor.dev"
    }
  ]
}
```

### Multiple Concurrent Policies

```json
{
  "halos_version": "0.3",
  "governance": [
    {
      "policy": "Meridian AI Usage Policy for Shared Infrastructure",
      "version": "2.1",
      "url": "https://internal.meridian.example/policies/ai-usage-shared-infra"
    },
    {
      "policy": "PCI-DSS",
      "version": "4.0",
      "url": "https://www.pcisecuritystandards.org/document_library/"
    },
    {
      "policy": "HALOS",
      "version": "1.0",
      "url": "https://halos.northharbor.dev"
    }
  ]
}
```

### Regulatory + Internal Policy

```json
{
  "halos_version": "0.3",
  "governance": [
    {
      "policy": "Lakeview WTP Change Management Policy",
      "version": "2024-rev3",
      "url": "https://lakeviewwtp.example/policies/change-management"
    },
    {
      "policy": "State DEQ Operating Permit",
      "version": "WTP-2019-0847",
      "url": "https://deq.state.example.gov/permits/wtp-2019-0847"
    }
  ]
}
```
